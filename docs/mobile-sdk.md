# 移动端桥接SDK设计

## 1. 移动设备的角色

```
角色1: 桥接中继 (Bridge)
  机器人 ◄──BLE──► 手机 ◄──WebSocket──► 主脑
  手机只做协议转发, 不做逻辑处理

角色2: 本地处理器 (Local Processor)
  机器人 ◄──BLE──► 手机 ──► 本地AI推理
  手机做ASR/NLU, 结果通过BLE下发机器人

角色3: 第三方代理 (API Proxy)
  机器人 ◄──BLE──► 手机 ──► Claude/GPT API
  手机封装第三方API调用, 转换为BCP指令

角色4: 控制面板 (Control Panel)
  用户通过手机APP直接控制机器人
  UI操作 → BCP指令 → BLE → 机器人
```

## 2. 技术选型

```
iOS:
  语言: Swift 5.9+
  UI框架: SwiftUI
  BLE: CoreBluetooth
  网络: URLSessionWebSocketTask

  语言: Kotlin 1.9+
  UI框架: Jetpack Compose
  BLE: Android BLE API (或 Nordic BLE库)
  网络: OkHttp WebSocket
```


## 4. BLE通信层

### 4.1 GATT服务交互

```
扫描过滤:
  Service UUID = 0xCB00
  设备名前缀 = "CB_" + 机器码后4位

连接流程:
  1. 扫描发现设备
  2. 连接 → MTU协商 (请求247字节)
  3. 发现服务 0xCB00
  4. 订阅 BCP_RX (0xCB002) Notify
  5. 读取 DEVICE_INFO (0xCB04) → 获取机器码/能力集/固件版本
  6. 写入 BCP_CONTROL → 告知手机端身份 (用户token)

数据通道:
  写入: BCP_TX (0xCB01) — 手机→机器人
  通知: BCP_RX (0xCB02) — 机器人→手机
```

### 4.2 BLE分片处理

```
BLE MTU = 247字节, 但BCP帧最大1024字节, 需要分片:

发送 (手机→机器人):
  BCP帧 > MTU-3 → 分片
  每片: [分片头(1B) + 数据(NB)]
  分片头: bit7=是否最后一片, bit6-0=分片序号(0-127)

接收 (机器人→手机):
  收到分片 → 缓冲 → 收到最后一片 → 重组 → BCP解析

超时: 单帧所有分片必须在2s内收齐, 否则丢弃
```

### 4.3 BLE连接保活

```
问题: BLE连接空闲时容易被系统杀死 (尤其iOS)

策略:
  - 每3s发送空帧 (BCP帧, CmdCount=0) 保持连接
  - iOS: 后台模式声明 "Uses Bluetooth LE accessories"
  - Android: 前台服务 + 通知栏常驻

断连重连:
  - 自动重连, 间隔 1s → 2s → 4s → 8s (最大)
  - 重连后重新订阅Notify
  - 未发送的BCP帧从队列重发
```

## 5. WebSocket通信层

### 5.1 连接管理

```
连接地址: ws(s)://主脑地址/ws?token={用户JWT}

连接选项:
  - 心跳间隔: 30s (WebSocket Ping)
  - 重连策略: 指数退避 1s → 30s
  - 证书固定: 硬编码CA指纹

消息格式:
  二进制帧 = BCP帧 (与机器人端完全一致)
  文本帧 = JSON控制消息 (连接状态/错误)
```

### 5.2 双通道选择

```
场景: 手机同时连接机器人(BLE)和主脑(WS)

数据流向:

  上行 (机器人→主脑):
    机器人 ──BLE──► 手机 ──WS──► 主脑
    手机原样转发BCP帧, 不解析payload

  下行 (主脑→机器人):
    主脑 ──WS──► 手机 ──BLE──► 机器人
    手机原样转发BCP帧

  本地处理模式:
    主脑 ──WS──► 手机 ──本地AI──► 手机 ──BLE──► 机器人
    手机拦截特定BCP指令, 本地处理后生成新指令下发
```

## 6. 移动端边缘计算

手机不只是桥接中继, 它有CPU/GPU/NPU, 可以分担主脑的工作。

### 6.2 语音处理管线

```
机器人 ──BLE(音频流)──► 手机

手机端处理:
  ┌──────────────────────────────────────────────────────────┐
  │  接收Opus音频流                                           │
  │       ↓                                                   │
  │  解码Opus → PCM                                           │
  │       ↓                                                   │
  │  噪声抑制 (WebRTC NS, 手机端运行)                         │
  │       ↓                                                   │
  │  本地ASR (iOS: Speech / Android: ML Kit)                  │
  │       ↓                                                   │
  │  ┌─────────────────┬─────────────────┐                    │
  │  │ 简单意图?       │ 复杂意图?       │                    │
  │  │ 本地NLU处理     │ 转发主脑处理    │                    │
  │  │ <500ms          │ 1-3s            │                    │
  │  └────────┬────────┴────────┬────────┘                    │
  │           ↓                 ↓                             │
  │  本地TTS              主脑回复→TTS                        │
  │       ↓                                                   │
  │  编码Opus → BLE下发机器人播放                              │
  └──────────────────────────────────────────────────────────┘

本地NLU支持的意图 (200+关键词):
  - 时间查询: "几点了" "今天几号"
  - 天气查询: "天气" "温度" (调用手机天气API)
  - 设备控制: "前进" "后退" "停下" "转圈"
  - 情绪表达: "笑一个" "生气" "开心"
  - 音量控制: "大声点" "小声点" "静音"
  - 系统命令: "充电" "重启" "关机"
  - 闲聊: "你好" "再见" "谢谢" (固定回复)

延迟对比:
  本地处理: ~500ms (ASR) + ~100ms (NLU) + ~200ms (TTS) ≈ 800ms
  主脑处理: ~500ms (传输) + ~1s (ASR) + ~1s (LLM) + ~500ms (传输) ≈ 3s
  第三方API: ~500ms (传输) + ~2s (API) + ~500ms (传输) ≈ 3s
```

### 6.3 视觉处理

```
机器人 ──BLE(图像快照)──► 手机

手机端处理:
  ┌──────────────────────────────────────────────────────────┐
  │  接收JPEG图像                                             │
  │       ↓                                                   │
  │  预处理: 缩放/裁剪/归一化                                 │
  │       ↓                                                   │
  │  ┌─────────────────────────────────────────────────────┐ │
  │  │  并行推理管线:                                       │ │
  │  │                                                      │ │
  │  │  人脸检测 (iOS Vision / Android ML Kit)  ~30ms      │ │
  │  │    → 检测到人脸 → 生成舵机追踪指令                  │ │
  │  │                                                      │ │
  │  │  物体识别 (Core ML / TFLite MobileNet)  ~50ms       │ │
  │  │    → 识别结果 → 生成描述文本                        │ │
  │  │                                                      │ │
  │  │  场景分类 (Core ML / TFLite)  ~40ms                 │ │
  │  │    → 室内/室外/人/宠物 → 触发对应行为               │ │
  │  │                                                      │ │
  │  │  手势识别 (Core ML / MediaPipe)  ~60ms              │ │
  │  │    → 挥手/指向/竖拇指 → 生成交互指令                │ │
  │  └─────────────────────────────────────────────────────┘ │
  │       ↓                                                   │
  │  结果汇总 → BCP指令下发 + 上报主脑                        │
  └──────────────────────────────────────────────────────────┘

视觉伺服 (手机端闭环):
  1. 手机检测到人脸位置 (x, y, 置信度)
  2. 计算偏移量: dx = target_x - center_x
  3. 生成舵机指令: pan += dx * gain
  4. BLE下发 SERVO_SET 指令
  5. 机器人执行 → 新图像 → 重复
  闭环频率: ~10Hz (受限于BLE图像传输)
```

### 6.4 传感器数据处理

```
机器人 ──BLE(原始传感器数据)──► 手机

手机端处理管线:
  ┌──────────────────────────────────────────────────────────┐
  │  接收传感器数据流                                         │
  │       ↓                                                   │
  │  ┌─────────────────────────────────────────────────────┐ │
  │  │  数据清洗:                                          │ │
  │  │  - 去除异常值 (3σ原则)                              │ │
  │  │  - 低通滤波 (去除高频噪声)                          │ │
  │  │  - 插值补全 (BLE丢包导致的缺失)                     │ │
  │  └─────────────────────────────────────────────────────┘ │
  │       ↓                                                   │
  │  ┌─────────────────────────────────────────────────────┐ │
  │  │  特征提取:                                          │ │
  │  │  - IMU: 步态检测、跌倒检测、手势识别               │ │
  │  │  - 环境: 趋势分析 (温度在上升?)、异常检测           │ │
  │  │  - ToF: 障碍物距离趋势、空间建模                   │ │
  │  │  - 触觉: 触摸模式识别 (轻触/长按/滑动)             │ │
  │  └─────────────────────────────────────────────────────┘ │
  │       ↓                                                   │
  │  ┌─────────────────────────────────────────────────────┐ │
  │  │  事件触发:                                          │ │
  │  │  - 跌倒事件 → 紧急停止 + 告警                      │ │
  │  │  - 温度异常 → 通知用户                              │ │
  │  │  - 长时间无互动 → 触发主动对话                      │ │
  │  │  - 检测到有人靠近 → 打招呼行为                      │ │
  │  └─────────────────────────────────────────────────────┘ │
  │       ↓                                                   │
  │  处理后的数据:                                            │
  │  - 实时数据 → 更新UI                                      │
  │  - 聚合数据 → 上报主脑 (每分钟一次, 而非每秒)            │
  │  - 异常事件 → 立即上报主脑 + 通知用户                    │
  └──────────────────────────────────────────────────────────┘

数据压缩效果:
  原始: 10Hz × 20字节 × 10传感器 = 2000 字节/秒
  处理后: 1Hz × 50字节 (聚合) + 事件 = ~100 字节/节
  节省: ~95% 带宽
```

### 6.8 第三方API代理

```
手机作为API代理, 解决机器人无法直接访问大模型的问题:

流程:
  机器人 ──BLE(音频/文本)──► 手机 ──HTTPS──► Claude/GPT API
  手机 ◄──BLE(回复)──────── 手机 ◄──HTTPS─── API响应

API封装:
  1. 收到机器人音频流 → 本地ASR转文本
  2. 构建对话上下文 (本地缓存最近10轮)
  3. 调用Claude/GPT API
  4. 回复文本 → TTS → 音频 → BLE下发

智能路由:
  简单意图 (本地关键词匹配) → 本地处理, <1s
  中等意图 (需要常识) → 调用小模型 (如GPT-4o-mini), 1-2s
  复杂意图 (需要推理) → 调用大模型 (如Claude/GPT-4o), 2-5s

意图复杂度判断:
  - 关键词命中 → 简单
  - 句子长度<10 + 无问号 → 简单
  - 含"为什么""怎么""解释" → 复杂
  - 其他 → 中等

上下文管理:
  - 本地SQLite存储对话历史
  - 每次API调用携带最近10轮上下文
  - 上下文总token控制在4000以内
  - 超限时摘要压缩旧对话
```

```
│                                                          │
│  BLE带宽:                                                │
│    正常: 图像+音频+传感器                                │
│    拥塞: 仅音频+控制指令, 图像降分辨率                  │
│    严重拥塞: 仅控制指令                                  │
└──────────────────────────────────────────────────────────┘
```

### 6.10 计算任务上报

```
手机处理的结果需要同步给主脑, 保持状态一致:

上报策略:
  实时事件 (立即上报):
    - 人脸检测结果 (有人/无人)
    - 跌倒/碰撞事件
    - 用户手动指令
    - 行为状态切换

  聚合上报 (每分钟):
    - 传感器数据摘要 (平均值/最大/最小)
    - 语音识别结果统计 (成功/失败次数)
    - 视觉检测统计 (检测到人脸次数)
    - 本地处理的任务数

  上线批量上报:
    - 离线期间的任务执行记录
    - 离线期间的对话历史
    - 离线期间的传感器异常事件

上报格式:
  BCP EVENT指令, payload为JSON:
  {
    "source": "mobile",
    "type": "sensor_aggregate",
    "timestamp": 1716633600,
    "data": {
      "temp_avg": 24.5,
      "temp_max": 26.1,
      "face_detected_count": 12,
      "voice_commands": 5
    }
  }
```

## 8. 数据流示意

```
完整数据流 (桥接模式):

  用户按住说话
       ↓
  手机录音 (16kHz PCM)
       ↓
  Opus编码
       ↓
  BCP封装: AUDIO_STREAM指令
       ↓
  BLE分片发送 → 机器人
       ↓
  机器人: 播放音频 (可选, 用于回声)
       ↓

  同时:
  手机本地ASR → 文本
       ↓
  BCP封装: TTS_TEXT指令 (如果本地处理)
       ↓
  BLE发送 → 机器人显示文本

  或:
  手机转发音频到主脑 (WebSocket)
       ↓
  主脑ASR → NLU → 回复
       ↓
  主脑下发TTS音频 (WebSocket)
       ↓
  手机转发 (BLE) → 机器人播放
```

## 9. 权限与安全

```
iOS权限:
  - NSBluetoothAlwaysUsageDescription (BLE后台)
  - NSLocalNetworkUsageDescription (局域网发现)
  - NSMicrophoneUsageDescription (语音输入)
  - NSCameraUsageDescription (扫码配对)

Android权限:
  - BLUETOOTH_SCAN + BLUETOOTH_CONNECT (BLE)
  - RECORD_AUDIO (语音输入)
  - CAMERA (扫码配对)
  - ACCESS_FINE_LOCATION (BLE扫描需要)
  - FOREGROUND_SERVICE (后台BLE)

安全:
  - BLE配对使用LESC (配对码确认)
  - WebSocket使用TLS + 用户JWT
  - 本地存储的token用Keychain(iOS)/KeyStore(Android)加密
  - 音频/图像不持久化 (除非用户主动保存)
```
